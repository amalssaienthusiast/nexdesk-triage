# High Level Design (HLD)

## 1. Introduction
The NexDesk Helpdesk OpenEnv project is a highly simulated, stateful web service deployed in a Dockerized environment designed for evaluating Large Language Model (LLM) agents. The system mimics a real IT operations queue where tickets must be prioritized, routed, and resolved within specific time limits under dynamic pressure levels.

## 2. Logical Architecture

The system operates across three tiers:
1. **Interface Tier:** Represented by the FastAPI container (`app.py`), the Pydantic type validators (`models.py`), and the MCP (Model Context Protocol) endpoints. This tier translates LLM JSON requests into internal Python state commands and broadcasts schemas.
2. **Simulation Tier:** Represented by `environment.py` and `graders.py`. Functioning as the core environment mechanics, it runs the state machine. It handles multi-step action aggregation (for tasks like `ticket_resolve` needing 3 individual steps), memory management via `accumulated_actions`, dynamic scaling of stress metrics based on randomized queue depths, and complex metric aggregations.
3. **Data Tier:** Ground truth for evaluations lies within `tickets.py`, functioning as an in-memory database of multi-dimensional test tickets covering varied scenarios across high, medium, and critical SLA urgencies. Metrics are temporarily housed in an in-memory dictionary instance bound by a Thread Lock.

## 3. Communication Patterns

- **Synchronous REST Protocol:** LLMs and Evaluators interact exclusively through REST APIs (`/reset`, `/step`). 
- **Thread Security & Mutexes:** Since multiple agents might try to hit the environment simultaneously during a broader validation check, the `NexDeskEnv` is fully synchronized using `threading.RLock()` to prevent race conditions during session manipulations.
- **Fail-Safe Processing:** Request payloads that fail the strict Pydantic definitions drop out immediately with `HTTP 422`.

## 4. Key Subsystems

### 4.1 Orchestration Platform
The execution pipeline is validated and run initially mostly via `inference.py`. The LLM client runs an orchestrator looping through the tasks, providing multi-shot learning contexts in system prompts to ensure the LLM knows how to format its output to properly interface with the API. It leverages retry logic for malformed JSON outputs generated by the LLM. 

### 4.2 SLA & Penalty Engine
To simulate time-passing in reality without holding the thread open, the start time is anchored structurally within the session payload during the `/reset` event. `elapsed_minutes` dynamically checks the epoch time on the incoming `/step` request to measure how much real time elapsed since reset, then mathematically applies a sliding penalty for actions taken post-SLA breach.

### 4.3 Crisis Batch Processing
The `crisis_surge` module bypasses the standard 1-ticket-per-session rule. It injects up to 10 sorted tickets into the environment sequence. The system actively clears the `accumulated` dictionary after every loop iteration so that each ticket is evaluated cleanly without polluting the agent state with earlier classifications. 

## 5. Deployment Architecture 
The service is explicitly containerized logic designed to drop smoothly into Huggingface Spaces or OpenEnv instances.
- **Base image:** Generic lightweight Python base
- **Dependencies:** Exclusively `fastapi`, `uvicorn`, `pydantic`, `requests` — minimizing weight.
- **Network Mapping:** Designed to expose internal logic securely over port `7860`.
